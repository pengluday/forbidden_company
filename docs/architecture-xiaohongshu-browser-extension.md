# Architecture: 小红书浏览器插件采集器

## 1. 架构目标
这套方案的目标不是做中心化代抓，而是做“用户浏览器内的采集入口 + 本机采集核心”。

核心原则：
- 采集动作必须由用户主动触发
- 真正的抓取逻辑尽量留在用户本机
- 插件负责入口、状态展示和配置
- 后端只负责本地采集、导出、去重和可选同步
- 免费版和付费版共享同一套采集核心

## 2. 总体方案
推荐三层结构：

### 2.1 Chrome Extension
负责：
- 在小红书页面上提供按钮入口
- 读取当前页面 URL
- 收集用户选择的配置
- 发起采集任务
- 展示进度、结果、错误

### 2.2 本机 Companion Service
负责：
- 读取本机 Chrome 登录态或 cookie
- 触发页面级采集
- 评论分页
- 帖子正文提取
- 数据清洗、去重、落库
- 导出 CSV / JSON

### 2.3 可选云端服务
后续才需要：
- 账号体系
- 授权与配额
- 支付
- 同步用户项目和历史

云端不直接承担抓取任务。

## 3. 组件拆分
### 3.1 Extension 端
建议包含：
- `manifest.json`
- `popup.html / popup.ts`
- `content script`
- `background service worker`
- `options page`

职责说明：
- `popup`：用户入口，展示当前 URL、采集按钮、输出状态
- `content script`：读取页面信息，必要时补充 DOM 状态
- `background`：与本机服务通信，缓存任务状态
- `options`：配置本机服务地址、默认评论上限、导出格式

### 3.2 本机服务端
建议沿用现有 Node/Python 采集核心，新增一个本地 API 层：
- `POST /api/xhs/collect`
- `POST /api/xhs/refresh`
- `GET /api/xhs/status`
- `GET /api/xhs/export`
- `POST /api/xhs/config`

职责说明：
- 接收插件任务
- 处理分页采集
- 统一落库
- 返回结果摘要

### 3.3 数据层
沿用现有本地数据思路：
- SQLite：采集结果、任务状态、基础配置
- CSV：可导出或用于兼容现有流程
- JSON：给插件或前端做轻量展示

## 4. 数据流
### 4.1 单条采集
1. 用户在小红书页面点击插件按钮
2. 插件获取当前 URL
3. 插件调用本机服务
4. 本机服务使用当前浏览器会话或 cookie 抓取帖子和评论
5. 结果写入 SQLite / CSV
6. 插件展示条数、摘要、失败信息

### 4.2 单条刷新
1. 用户输入目标 URL
2. 插件发起 refresh 请求
3. 本机服务删除旧记录
4. 重新采集帖子和评论
5. 覆盖导出结果

### 4.3 导出
1. 用户点击导出
2. 插件请求本机服务生成文件
3. 服务返回下载链接或文件路径

## 5. 评论采集策略
### 5.1 采集原则
- 先抓帖子正文
- 再抓评论
- 评论必须支持分页
- `comment_limit=0` 表示抓取全部可见评论

### 5.2 技术策略
优先级建议：
1. 直接使用本机服务里的评论分页逻辑
2. 如果直连接口失败，使用浏览器会话回退
3. 如果当前登录态不可用，明确返回失败原因

### 5.3 风险控制
- 不做无限并发
- 不做后台静默批量抓取
- 不做自动刷新循环
- 所有请求都绑定到用户主动动作

## 6. 插件权限设计
### 必要权限
- `storage`：保存本地配置
- `activeTab`：读取当前页面上下文
- `scripting`：在页面中注入辅助脚本
- `tabs`：识别当前标签页 URL
- `host_permissions`：只允许小红书域名和本机服务域名

### 权限约束
- 默认只允许小红书相关域名
- 默认只允许访问本机服务地址
- 不申请与功能无关的高风险权限

## 7. 本机服务 API
建议最少实现以下接口：

### `POST /api/xhs/collect`
请求体：
```json
{
  "url": "https://www.xiaohongshu.com/discovery/item/...",
  "comment_limit": 0,
  "include_comments": true,
  "company_name": "",
  "source_platform": "小红书"
}
```

返回：
```json
{
  "ok": true,
  "post_count": 1,
  "comment_count": 128,
  "output_csv": "/path/to/file.csv",
  "records": [...]
}
```

### `POST /api/xhs/refresh`
请求体：
```json
{
  "url": "https://www.xiaohongshu.com/discovery/item/...",
  "comment_limit": 0
}
```

### `GET /api/xhs/status`
返回：
- 本机服务是否在线
- cookie 是否可用
- 当前默认评论上限

### `POST /api/xhs/config`
支持保存：
- 本机服务地址
- 默认评论上限
- 导出格式
- cookie 文件路径

## 8. 存储设计
### 8.1 SQLite
建议继续沿用 `collected_evidence`，并补充：
- `collection_jobs`
- `collection_runs`
- `plugin_settings`

### 8.2 本地配置
建议保存到：
- 插件 `chrome.storage.local`
- 本机服务 `data/` 目录

### 8.3 导出文件
建议按日期命名：
- `source-intake-xiaohongshu-YYYY-MM-DD.csv`
- `source-intake-xiaohongshu-YYYY-MM-DD.json`

## 9. 免费版与付费版边界
### 免费版
- 单条页面采集
- 基础评论分页
- 导出 CSV / JSON
- 本地保存
- 手动触发

### 付费版
- 批量任务队列
- 历史项目管理
- 多工作区
- 同步与备份
- 使用额度

### 设计约束
免费版和付费版必须共享：
- 同一套数据模型
- 同一套采集核心
- 同一套去重逻辑

## 10. 合规与风控
- 不在文案中承诺规避平台风控
- 不以“批量绕过”作为产品卖点
- 不在插件里内置超高频自动任务
- 明确提示用户对自己的操作负责
- 保留来源链接与采集时间，便于追溯

## 11. 性能与可靠性
目标：
- 用户点击后 2 秒内看到开始状态
- 单条采集结果在可接受时间内返回
- 评论分页必须可中断、可重试

建议：
- 采集任务异步化
- 插件只展示状态，不直接阻塞页面
- 服务端记录任务状态，便于恢复和调试

## 12. 失败模式
### 12.1 cookie 失效
表现：
- 评论接口返回账号异常

处理：
- 提示用户重新登录
- 提供刷新 cookie 的入口

### 12.2 页面结构变化
表现：
- 帖子正文或评论解析失败

处理：
- 降级为只保留来源与标题
- 记录失败日志

### 12.3 本机服务不可用
表现：
- 插件连不上本机端口

处理：
- 提示启动本机服务
- 显示配置页

## 13. 迭代路线
### Phase 1
- 插件 UI
- 本机服务接入
- 单条采集
- 评论分页
- 导出

### Phase 2
- 最近任务列表
- 失败重试
- 配置同步
- 更清晰的本地状态管理

### Phase 3
- 免费/付费账号体系
- 项目管理
- 批量采集
- 同步与团队能力

## 14. 推荐技术栈
- 插件：Manifest V3 + TypeScript
- 本机服务：Node.js + Python 现有采集脚本
- 存储：SQLite + CSV
- 导出：本地文件下载
- UI：尽量保持轻量，不引入不必要的框架

## 15. 结论
这不是一个“云端爬虫产品”，而是一个“用户浏览器里的采集工具”。

只要边界守住：
- 插件做入口
- 本机做采集
- 云端只做账号和商业化

就能同时满足：
- 免费版快速验证
- 付费版可扩展
- 风控压力分散
- 复用你现有的采集资产
